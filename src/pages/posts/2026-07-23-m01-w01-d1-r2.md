# 다음 token 예측을 데이터 경로로 보기

> curriculum ref: **M01-W01-D1** · 작성일: 2026-07-22
> 이 글은 공개 출처로 확인 가능한 내용과 개인 해석, 그리고 toy 가정을 분리해서 씁니다.

---

## 오늘의 질문

**LLM이 다음 token 하나를 고르는 순간, 실제로는 어떤 data movement가 일어나는가?**

보통 "다음 token 예측"은 확률 문제로 설명된다. 이전 token을 보고 다음 token의 확률분포를 만들고, 거기서 하나를 뽑는다. 여기까지는 수식 이야기다.

그런데 하드웨어 관점에서 보면 같은 문장이 이렇게 바뀐다.

> token 하나를 뽑기 위해 **약 15 GB의 weight를 HBM에서 SRAM으로 끌어오고**, 그중 실제 산술 연산은 **약 17 GFLOP** 밖에 안 한다.

이 비대칭이 오늘의 주제다. 목표는 용어 정의가 아니라 **하나의 system behavior**를 설명하는 것이다. 그 behavior는 이것이다.

> **decode 단계에서 embedding table과 lm_head는 shape가 완전히 같은데, 메모리 접근 패턴이 정반대다.**
> 하나는 capacity를 먹지만 거의 읽히지 않고, 하나는 매 step 전부 읽힌다.

---

## 핵심 개념

다섯 개 용어를 "무엇인가"가 아니라 **"어떤 메모리 접근을 유발하는가"**로 정리한다.

| 개념 | 교과서적 정의 | 데이터 경로에서의 정체 |
|---|---|---|
| **token** | 텍스트를 자른 최소 단위 | 사실상 **정수 인덱스 하나(4 B)**. 그 자체는 데이터가 아니라 주소다. |
| **embedding** | token을 벡터로 매핑 | `[V, d]` 행렬에 대한 **random row lookup**. gather 연산이고, 전체 행렬 중 한 줄만 건드린다. |
| **logit** | vocabulary 전체에 대한 점수 | hidden state와 `[d, V]` 행렬의 GEMV 결과. **출력 하나(500 KiB) 만들려고 1 GB를 읽는다.** |
| **softmax** | logit → 확률분포 | vocabulary 축 전체를 훑는 reduction. 연산량은 작지만 **V 길이 벡터를 최소 2번 왕복**한다. |
| **autoregressive decode** | 한 번에 한 token씩 생성 | 앞의 전 과정이 **token 개수만큼 직렬 반복**된다. 병렬화가 안 되는 축이 생긴다. |

**확정 사실:** Transformer 계열 decoder가 이전 출력을 다시 입력으로 넣어 한 token씩 생성한다는 구조, 그리고 입력 embedding 행렬과 출력 projection 행렬이 같은 shape를 가진다는 점은 원 논문에 기술되어 있다 (Vaswani et al., 2017, §3.2·§3.4).

**내 해석:** 논문은 이 구조를 *모델링* 관점에서 설명하지만, "한 token씩 직렬"이라는 성질이 하드웨어에서는 **batch=1 decode를 memory-bound로 만드는 근본 원인**이 된다. 이건 논문이 다루는 주제가 아니라 시스템 쪽 해석이다.

## 계산과 표

### toy 설정 (실제 제품 스펙 아님, 계산용 가정)

실제 상용 모델 스펙을 인용하지 않고, 8B급 모델을 흉내 낸 **toy config**를 쓴다.

```
d_model     = 4096
n_layers    = 32
n_heads     = 32,  head_dim = 128
n_kv_heads  = 8    (GQA)  → kv_dim = 1024
d_ff        = 14336 (SwiGLU, 3개 행렬)
V           = 128,000
dtype       = bf16 (2 B)
context S   = 4096
batch       = 1
```

예문: `"메모리 병목은 계산보다 자주 문제가 된다"` → **12 token으로 토크나이즈되었다고 가정**한다 (토크나이저마다 달라지므로 이건 가정이다).

### 1) tensor shape table (prefill 12 token 기준)

| 단계 | shape | dtype | 바이트 |
|---|---|---|---|
| token ids | `[1, 12]` | int32 | 48 B |
| embedding weight | `[128000, 4096]` | bf16 | 1.049 GB |
| embedding output | `[1, 12, 4096]` | bf16 | 98,304 B (96 KiB) |
| layer별 hidden state | `[1, 12, 4096]` | bf16 | 96 KiB |
| KV cache (prefill 후) | `[32, 2, 12, 8, 128]` | bf16 | 1.57 MB |
| logits (마지막 위치만) | `[1, 1, 128000]` | fp32 | 512,000 B (500 KiB) |
| probs | `[1, 1, 128000]` | fp32 | 500 KiB |
| next token id | `[1, 1]` | int32 | 4 B |

여기서 이미 이상한 점이 보인다. **입력 48 B → 출력 4 B**인데, 중간에 GB 단위 행렬이 두 번 등장한다.

### 2) 메인 계산표 — decode 1 step, batch=1

| 항목 | 조건/shape | 계산식 | 결과 | 해석 |
|---|---|---|---|---|
| embedding table 용량 | `[128000, 4096]` bf16 | `128000 × 4096 × 2` | **1.049 GB** | HBM capacity를 상시 점유 |
| embedding 실제 읽기 | 1 row | `4096 × 2` | **8,192 B (8 KiB)** | 테이블의 **0.00078%**만 읽음 → capacity-bound |
| layer당 attn weight | Wq,Wk,Wv,Wo | `4096² + 4096×1024 ×2 + 4096²` | 41.94 M param | GQA로 K/V가 1/4 |
| layer당 MLP weight | gate/up/down | `4096×14336 × 3` | 176.16 M param | MLP가 layer 파라미터의 **81%** |
| transformer weight 전체 | 32 layers | `(41.94 + 176.16)M × 32 × 2 B` | **13.96 GB** | 매 step 전량 읽힘 |
| lm_head 읽기 | `[4096, 128000]` | `4096 × 128000 × 2` | **1.049 GB** | embedding과 같은 크기, **100% 읽힘** |
| KV cache 읽기 | S=4096, GQA 8 | `2 × 32 × 4096 × 1024 × 2` | **536.9 MB** | context 길이에 **선형 비례** |
| KV cache 증가율 | token 1개당 | `2 × 32 × 1024 × 2` | **131,072 B (128 KiB/token)** | 32K context면 4.2 GB |
| **decode 1 step 총 HBM traffic** | 위 합계 | `13.96 + 1.049 + 0.537 + ~0` | **≈ 15.55 GB** | 이게 오늘의 핵심 숫자 |
| decode 1 step FLOP | weight GEMV + attention | `2 × 7.50G + 2.15G` | **≈ 17.1 GFLOP** | |
| **arithmetic intensity** | FLOP ÷ byte | `17.1e9 ÷ 15.55e9` | **≈ 1.10 FLOP/B** | 극단적으로 낮음 |
| machine balance (가정) | 989 TFLOPS ÷ 3.35 TB/s | `989e12 ÷ 3.35e12` | **≈ 295 FLOP/B** | 가속기가 요구하는 균형점 |
| **불균형 배율** | 295 ÷ 1.10 | — | **≈ 268×** | roofline상 완전한 memory-bound |
| latency 하한 (대역폭) | 3.35 TB/s 가정 | `15.55e9 ÷ 3.35e12` | **≈ 4.64 ms/token** | ≈ **215 token/s** 상한 |
| latency 하한 (연산) | 989 TFLOPS 가정 | `17.1e9 ÷ 989e12` | **≈ 17.3 µs/token** | 연산 유닛은 **99.6% 놀고 있음** |
| 에너지 (거친 추정) | 4 pJ/bit = 32 pJ/B 가정 | `15.55e9 × 32e-12` | **≈ 0.50 J/token** | 데이터 이동만 계산한 값 |
| 비교: prefill 12 token | weight는 1회만 읽음 | `205 GFLOP ÷ 15.0 GB` | **≈ 13.7 FLOP/B** | decode보다 **12배** 효율 |
| 비교: decode batch=32 | weight 공유 | `547 GFLOP ÷ 17.6 GB` | **≈ 31 FLOP/B** | batching이 곧 대역폭 절약 |

> 표의 GB는 10진(10⁹) 기준이다. GiB(2³⁰)로 쓰면 13.96 GB = 13.0 GiB다. 벤치마크 비교할 때 이 단위 혼동이 자주 사고를 낸다.

## 데이터 경로 또는 메모리 병목

### 관찰 1 — 같은 shape, 정반대 접근 패턴

`embedding`과 `lm_head`는 둘 다 `128000 × 4096`, 둘 다 1.049 GB다. 그런데:

- **embedding**: decode step마다 **8 KiB**만 읽는다 (한 row). random gather. 대역폭은 거의 안 쓰고 **capacity만 잡아먹는다.**
- **lm_head**: decode step마다 **1.049 GB 전부** 읽는다. sequential GEMV. **step당 전체 traffic의 6.7%**를 혼자 차지한다.

같은 크기의 두 행렬이 시스템에 거는 부하의 성격이 완전히 다르다. 이건 "행렬 크기"만 보는 파라미터 카운트로는 절대 안 보인다.

**내 해석:** 그래서 vocabulary를 키우는 결정은 파라미터 수보다 **decode latency에 더 직접적으로 비싸다.** V를 2배로 하면 embedding은 capacity만 2배가 되지만 lm_head는 step당 traffic이 그대로 2배가 된다. (검증 필요한 해석이다.)

### 관찰 2 — 병목의 정체는 "재사용 없음"

batch=1 decode의 본질은 **weight 하나를 읽어서 곱셈 한 번 하고 버린다**는 것이다. bf16 행렬 원소 1개(2 B)당 연산은 곱셈 1 + 덧셈 1 = 2 FLOP. 그래서 intensity가 구조적으로 **1 FLOP/B 근처에 고정**된다. 알고리즘을 바꾸지 않는 한 이 값은 안 움직인다.

반면 가속기는 295 FLOP/B를 요구한다. 이 간극을 메우는 방법은 셋뿐이다.

1. **byte를 줄인다** → quantization (bf16 → int8/int4). intensity 2~4배.
2. **FLOP을 늘린다(재사용)** → batching, speculative decoding. batch B면 intensity ≈ B배.
3. **대역폭을 늘린다** → HBM 세대 교체, 혹은 메모리를 연산 가까이 옮기기.

**내 해석:** 최근 몇 년간 LLM 서빙 최적화가 "더 빠른 커널"보다 **"어떻게든 weight를 덜 읽기"** 쪽으로 몰린 이유가 이 표 한 줄(1.10 vs 295)로 설명된다고 본다.

### 관찰 3 — KV cache는 시한폭탄

S=4096에서 KV cache traffic은 537 MB로 전체의 3.5%다. 무시할 만하다. 그런데 **token당 128 KiB씩 자란다.**

| context S | KV cache | step당 traffic 비중 |
|---|---|---|
| 4,096 | 0.54 GB | 3.5% |
| 32,768 | 4.29 GB | 22.2% |
| 131,072 | 17.18 GB | **53.4%** |

128K context에서는 **KV cache 읽기가 weight 읽기보다 커진다.** weight는 고정비, KV는 변동비인데 변동비가 고정비를 추월하는 지점이 있다.

**내 해석:** 이 교차점이 GQA/MQA, KV 압축, PagedAttention 같은 기법이 나온 지점이라고 이해하고 있다. 그리고 이 지점부터는 "모델을 어디에 두느냐"가 아니라 **"세션 상태를 어디에 두느냐"**가 시스템 설계 문제가 된다. 여기서 CXL 같은 tiered memory 논의로 이어질 것 같은데, 오늘은 계산까지만 하고 결론은 미룬다.

## 확정 사실 / 해석 / 추정

### 확정 사실 (공개 출처로 확인 가능)
- Transformer decoder는 이전에 생성한 심볼을 다음 step의 입력으로 소비하는 auto-regressive 구조다. (Vaswani et al., 2017)
- 출력단은 선형 변환 후 softmax로 다음 token 확률을 만든다. 이때 입력 embedding 행렬과 출력 선형 변환 행렬은 같은 shape이며, 논문은 두 층이 가중치를 공유하는 구성을 기술한다. (같은 논문 §3.4)
- 표에 쓴 산술(파라미터 수, 바이트 수, 나눗셈)은 위에 명시한 toy config로부터 결정론적으로 계산된 값이다. 계산식을 표에 그대로 적어두었으므로 재현 가능하다.

### 해석 (내가 계산과 구조에서 끌어낸 것, 출처 없음)
- decode의 arithmetic intensity가 1 FLOP/B 근처에 **구조적으로 고정**된다는 점. bf16 원소당 2 FLOP이라는 산술에서 바로 나온다.
- embedding은 capacity 문제, lm_head는 bandwidth 문제라는 **역할 분리**.
- V(vocabulary) 확대가 파라미터 수 증가보다 decode latency에 더 직접적으로 비싸다는 주장.
- KV cache가 특정 context 길이(위 toy config에서는 100K 부근)를 넘으면 weight traffic을 추월한다는 교차점의 존재.

### 추정 (검증 안 됨, 그대로 인용하면 안 됨)
- HBM 대역폭 3.35 TB/s, bf16 dense 989 TFLOPS라는 숫자. 실제 가속기 등급을 흉내 낸 값이지만 **벤더 datasheet로 확인하지 않았다.** 이 두 값이 바뀌면 268배, 4.64 ms, 215 token/s가 전부 바뀐다.
- 메모리 접근 에너지 4 pJ/bit(=32 pJ/B). 자릿수 감각용이며 세대/구성에 따라 크게 달라진다. 0.50 J/token은 **오차 범위가 몇 배**라고 봐야 한다.
- 예문이 12 token으로 쪼개진다는 가정. 토크나이저에 따라 다르고, 한국어는 특히 편차가 크다.
- decode에서 weight가 정확히 1회만 읽힌다는 가정. 실제로는 커널 구현·캐시 재사용·kernel fusion에 따라 달라지므로 이 15.55 GB는 **이상적 하한**이다.

## 다음 질문

오늘 계산이 남긴, 아직 답 못 한 것들:

1. **15.55 GB는 정말 하한인가?** 실제 커널은 weight를 몇 번 읽는가. L2/SRAM 재사용을 넣으면 이 숫자가 얼마나 내려가는지 프로파일러로 확인해야 한다.
2. **KV cache 교차점을 일반화하면?** `L, S, kv_dim`을 변수로 두고 "KV traffic > weight traffic"이 되는 S를 닫힌 식으로 유도해보기.
3. **quantization은 정확히 몇 배 이득인가?** int8/int4로 갔을 때 intensity와 latency가 어떻게 움직이는지, 그리고 KV cache는 별도로 양자화해야 하는지.
4. **capacity vs bandwidth 분리가 하드웨어 계층으로 이어지는가?** embedding table처럼 "크지만 거의 안 읽히는" 데이터는 HBM이 아니라 CXL 계층에 둘 수 있는가. latency penalty가 gather 패턴을 얼마나 망가뜨리는가. ← 이게 M01 후반에서 다룰 진짜 질문 같다.
5. **미확인 스펙 3개를 확정하기.** HBM 대역폭, bf16 TFLOPS, pJ/bit. 이 세 개를 datasheet로 못 박기 전까지 오늘 결론의 배율(268배)은 인용하면 안 된다.
