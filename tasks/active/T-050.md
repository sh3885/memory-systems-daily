# Objective

Remove the unmerged content-branch PR from the approved publishing path.
Commit approved post content directly to `main`, which is the branch that
triggers the production Pages workflow, then wait long enough to verify the
real post URL before reporting success.

# Handoff

- Files changed: `automation/publishing/github-app-publisher.mjs`, `automation/telegram/worker.mjs`, `automation/tests/publication-service.test.mjs`, and task records.
- Decisions: when the configured content branch is GitHub's default branch, the publisher writes the approved post directly and skips PR creation; the Worker waits up to roughly 55 seconds for deployment verification by default.
- Validation result: 101 automation tests passed; `scripts/check.ps1` passed with an Astro production build.
- Risks: GitHub Actions or Cloudflare Pages outages can still make verification fail, but a publication is now recorded only after the real post URL contains its title and claim ledger marker.
- Next action: configure `GITHUB_CONTENT_BRANCH=main`, deploy the Worker, then run `/publish-retry` for the active approved revision and verify the production URL.
