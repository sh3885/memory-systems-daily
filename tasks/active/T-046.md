# Objective

Review of the current manual `/prompt` output (built by `buildDraftPrompt` in
`automation/telegram/lesson-command-router.mjs` plus the context injected by
`buildDailyLessonPromptContext` / `buildArtifactScaffold` in
`automation/content/daily-lesson-prompts.mjs`) found instructions that can
actively hurt post quality, plus a missing decision rule the user specifically
asked for: a way to tell Claude *when a diagram is actually worth drawing*
instead of "add one if it helps" repeated with no concrete criteria.

Fix the four issues below, ordered by impact.

## 1. Persona sentence fights Rule #2 (highest impact)

`buildDraftPrompt` (automation/telegram/lesson-command-router.mjs:100) opens with:

> "너는 LLM, 컴퓨터 아키텍처, DRAM/HBM/CXL, 메모리 병목을 함께 설명하는 한국어 기술 튜터다."

But draft rule #2 in the same prompt (line ~110) says:

> "관련 없는 memory 병목이나 시스템 설명을 억지로 넣지 않는다."

These two lines pull in opposite directions on every single day whose
`theme` is `LLM` or `System` rather than `Memory` (see `dailyLessons[].theme`
in `automation/content/daily-lesson-prompts.mjs`). The persona primes Claude
to fold in memory-bottleneck framing regardless of topic; the rule then tells
it not to. Net effect: inconsistent behavior, and a real risk of forced,
unnatural memory tie-ins on pure-LLM-fundamentals days (e.g. `M01-W01-D1`,
"LLM은 무엇이고 왜 필요한가").

**Fix**: make the persona line theme-neutral, or derive it from
`lesson.theme` so it only leans into DRAM/HBM/CXL framing on days whose
`theme === "Memory"`. Remove "DRAM/HBM/CXL, 메모리 병목" from the default
persona sentence.

## 2. No concrete selection criteria for diagrams/tables (user's main ask)

Right now the "only add it if it helps" instruction is repeated near-verbatim
three times with no way for Claude to actually decide:

- quality gate: "계산, 표, 실험, 다이어그램은 설명을 더 좋게 할 때만 선택한다."
- `buildArtifactScaffold` (automation/content/daily-lesson-prompts.mjs:118):
  "다이어그램을 넣는다면 독자가 글만으로 이해하기 어려운 흐름이나 관계를
  보여 줄 때만 넣는다."
- draft rule #3: "계산, 표, 실험, 다이어그램은 설명을 더 분명하게 만들 때만
  넣는다."

Repeating the same soft constraint three times does not give Claude a
decision procedure, so it can still default to inserting a diagram out of
habit.

**Fix**: replace the three repeated lines with one consolidated block that
gives explicit positive/negative examples and a self-check question, e.g.:

```
다이어그램을 그리기 좋은 경우:
- 여러 단계를 순서대로 거치는 처리 흐름 (예: token -> embedding -> logit)
- 시간에 따라 달라지는 타임라인 (예: prefill과 decode 구간)
- 포함관계나 계층 구조 (예: L1/L2/L3 캐시 계층)
- 물리적으로 데이터가 이동하는 경로

다이어그램을 만들지 않는 경우:
- 개념 하나를 정의만 하면 되는 경우
- 단순 목록 나열이 bullet point로 이미 충분한 경우
- 인과관계나 순서가 없는 배경 설명
- 문장 한두 줄로 이미 이해되는 내용

먼저 "이 다이어그램이 없으면 독자가 이해 못 하는 게 무엇인가?"를 한 문장으로
답할 수 있을 때만 그린다. 답이 안 나오면 그리지 않는다.
```

Apply the same before/after treatment to table/calculation guidance if it
has the same repeat-with-no-criteria problem.

## 3. Possible duplicate Q&A section

`dailyLessons[].postSections` sometimes includes `"질문과 답변"` (e.g.
`M01-W01-D1`), while the fixed `출력 형식` template in `buildDraftPrompt`
separately mandates `## 자주 나올 질문과 답변`. There is no instruction
telling Claude whether these are the same section or two separate ones, so a
draft can end up with two near-identical Q&A blocks.

**Fix**: either drop `"질문과 답변"` from `postSections` entries where it
duplicates the fixed template section, or add one line clarifying they must
be merged into a single section.

## 4. (Lower priority, no urgent fix required) Fixed 7-header template ignores per-post-type structure

`src/data/curriculum.ts` already defines 5 `postBlueprints` with different
`requiredBlocks` per post type (Concept Deep Dive, Memory Traffic Note,
Interface Comparison, Experiment Log, Trend Radar). The runtime prompt in
`daily-lesson-prompts.mjs` never reads this — every day gets the exact same
7-heading skeleton (오늘의 질문 / 먼저 답하기 / 핵심 섹션 / 질문과 답변 /
확정 사실·해석·추정 / Claim ledger / 다음 질문) regardless of `postType`.
Flagging as a known gap; do not block this task on fixing it unless time
allows, since it's a larger structural change (would need
`daily-lesson-prompts.mjs` to select a template per `postType` and to stay in
sync with `postBlueprints` in `src/data/curriculum.ts`, which is a separate,
duplicated data source already tracked as a known risk in prior handoffs).
